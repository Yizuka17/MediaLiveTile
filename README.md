# Media Live Tile

一个面向 **Windows 10** 的媒体动态磁贴工具。  
它会读取系统当前的媒体会话（System Media Transport Controls / Global Media Control），并将正在播放或已存在的媒体信息显示到开始菜单磁贴中。
> 当前版本已经停止迭代，最多做小修小补。请关注新项目：https://github.com/Yizuka17/MediaLiveTile.Hybrid

## 项目说明

本项目主要是个人自用性质的实验项目，最初目标是补足自己在 Windows 10 上使用媒体应用时对动态磁贴显示的需求。

项目开发过程中大量借助了 AI 工具辅助完成，包括但不限于代码生成、结构整理、问题排查和功能迭代。因此，这不是一个按照工业规范长期维护的正式产品，更接近“个人需求驱动下持续打磨的工具”。

同时需要说明的是：

- 本项目主要面向个人兴趣与自用场景
- 功能更新与问题修复没有固定计划
- debug、维护和继续迭代基本随缘
- 对稳定性、兼容性和长期支持不做保证

如果你愿意自行研究、修改或扩展，欢迎参考源码。

## 功能特性

### 1. 读取系统媒体会话
应用可读取 Windows 10 系统当前的媒体会话信息，包括：

- 来源应用
- 标题
- 歌手
- 专辑
- 播放状态
- 缩略图 / 封面

当前已验证可识别如下来源中的一部分或全部信息：

- 网易云音乐 UWP
- Microsoft Edge
- Chrome（视系统媒体会话暴露情况）
- 其他已接入系统媒体控制接口的播放器或网页媒体

---

### 2. 动态磁贴显示当前媒体
支持更新应用主磁贴，并尽量适配以下磁贴尺寸：

- 小磁贴
- 中磁贴
- 宽磁贴
- 大磁贴

当前设计逻辑为：

- **小磁贴**：优先显示封面
- **中磁贴**：封面态 / 信息态切换
- **宽磁贴**：左侧封面，右侧文字信息
- **大磁贴**：封面态 / 信息态切换

其中：

- 封面优先显示
- 信息态显示标题、歌手、来源
- 无封面时可尝试使用来源应用图标兜底

---

### 3. 支持多个次级磁贴（Secondary Tile）
应用支持在界面中直接固定多个独立磁贴。  
每个次级磁贴都可以绑定不同的媒体槽位。

例如：

- 主媒体
- 次媒体#1
- 次媒体#2
- 次媒体#3
- 次媒体#4
- 次媒体#5

这意味着你可以在开始菜单上固定多个磁贴，让它们分别显示不同的媒体来源。

---

### 4. 固定槽位选择
与“根据当前会话数量动态变化”不同，本项目采用了**固定槽位**方案。  
每种磁贴尺寸都可以独立选择显示目标：

- 主媒体
- 次媒体#1
- 次媒体#2
- 次媒体#3
- 次媒体#4
- 次媒体#5

当目标槽位当前没有对应媒体时，会显示：

- 默认兜底图标
- 文案：`当前无媒体`

这样可以避免刷新后因媒体数量变化导致磁贴目标错乱的问题。

---

### 5. 无封面时应用图标兜底
当媒体会话没有提供封面时，应用会尝试使用来源应用图标作为封面替代：

优先级大致为：

1. 专辑封面
2. 系统可解析的应用图标
3. 项目内置的来源图标资源
4. 默认通用图标

这能明显改善“无封面媒体”时磁贴过于空白的问题。

---

### 6. 主页面实时预览
应用主页面提供磁贴预览区，可直接查看：

- 小磁贴预览
- 中磁贴封面 / 信息预览
- 宽磁贴预览
- 大磁贴封面 / 信息预览

这让磁贴显示逻辑更直观，也方便调试不同媒体槽位的展示效果。

---

### 7. 媒体会话监测
主页面可显示当前检测到的媒体会话摘要，包括：

- 主媒体
- 次媒体
- 前 4 个媒体会话列表
- 最后刷新时间

并支持：

- 手动刷新磁贴
- 自动刷新监测（当前为 30 秒）

## 当前实现思路

本项目当前仍基于 **纯 UWP** 架构，核心依赖如下能力：

- `GlobalSystemMediaTransportControlsSessionManager`
- UWP Live Tile / Tile Notification
- SecondaryTile
- 本地缓存封面与应用图标

整体思路是：

1. 读取系统媒体会话
2. 根据排序规则选择主媒体与候选媒体
3. 为各尺寸磁贴生成显示内容
4. 更新主磁贴与所有已固定的次级磁贴

---

## 媒体排序规则

当前媒体排序的主要思路为：

- 播放中优先
- 音乐类媒体优先
- 信息完整度更高的优先
- 当前系统会话优先

排序结果中：

- 第 1 项视为主媒体
- 第 2 项视为次媒体
- 之后依次作为候选媒体

## 系统要求

### 推荐环境
- Windows 10
- 支持系统媒体会话的播放器 / 浏览器 / 媒体应用

### 不推荐环境
- Windows 11

原因很简单：  
本项目以 **Win10 时代的动态磁贴** 为核心能力，而 Windows 11 对传统 Live Tile 的支持已经不再完整。

## 已知不足 / 当前限制

### 1. 仍受 UWP 动态磁贴系统限制
虽然应用已经尽量实现了所需效果，但 UWP Live Tile 本身存在一些先天限制，例如：

- 磁贴布局自由度有限
- 某些系统版本下磁贴队列切换节奏不可控
- 小磁贴动态显示在不同环境下表现不完全一致
- 初始固定后的次级磁贴默认尺寸通常由系统决定（一般是中磁贴）

也就是说，应用可以更新各尺寸磁贴内容，但**无法强制系统一开始就以某个指定尺寸固定磁贴**。

---

### 2. 某些媒体来源封面质量较低
应用读取的是系统媒体会话暴露出的缩略图。  
如果来源应用本身提供的是低分辨率或压缩过的封面，那么磁贴显示时也只能使用该图。

这在以下场景中尤其常见：

- 浏览器网页媒体
- 某些视频平台
- 某些没有完整暴露高质量封面的播放器

---

### 3. 应用图标兜底并非对所有来源都稳定
对于 UWP / 打包应用，应用图标通常可以较稳定获取；  
但对于部分传统桌面程序（如浏览器或第三方播放器），有时需要依赖项目内置图标资源做静态映射兜底。

因此“无封面 -> 应用图标”这条链路并不保证对所有来源 100% 完美一致。

---

### 4. 纯 UWP 后台能力有限
当前版本仍是 **纯 UWP**。  
这意味着它在后台运行能力上存在天然限制：

- 刷新频率和调度时机受系统影响
- 无法像传统桌面程序那样稳定长期常驻
- 无法自然实现最小化到系统托盘并长期驻留

这也是当前架构最大的现实瓶颈之一。

---

## 为什么未来会脱离纯 UWP

虽然纯 UWP 在“动态磁贴”这件事上很原生，但随着项目逐步完善，已经越来越明显地暴露出几个问题：

1. **后台调度不够自由**
2. **无法稳定常驻**
3. **无法最小化到系统托盘**
4. **与用户对“后台媒体助手”的预期存在差距**

因此，未来版本计划逐步从“纯 UWP”过渡到：

- 打包桌面应用
- 或 WinUI / 桌面桥接 / 混合架构

目标是实现：

### 未来方向
- 最小化到系统托盘
- 后台常驻运行
- 更稳定地监测系统媒体会话
- 更自由地控制刷新与磁贴更新
- 保留磁贴能力，同时获得桌面程序的后台能力

换句话说：

> 本项目未来将不再局限于纯 UWP，而是转向更适合“后台媒体磁贴助手”定位的桌面化方案。
